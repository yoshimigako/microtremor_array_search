# 詳細報告書: microtremor_array_search202501-v4_local_crs.py 解析

作成日: 2026-03-18
対象ファイル: `microtremor_array_search202501-v4_local_crs.py`

追記運用条件:

- 実行確認に使う想定環境は `conda activate pygmt`
- 今後の改修や検証は別ディレクトリで実施する
- Git 管理側の現行版にも GeoFabrik ローカル PBF 読み込み実装を取り込んだ

## 1. 要約

`microtremor_array_search202501-v4_local_crs.py` は、OpenStreetMap 上の道路・公園・水域・幹線道路/鉄道情報を使い、指定した中心点候補の周囲に「正三角形に近い 3 点観測アレイ」を自動設計するスクリプトである。

設計思想は次の 2 段階に分かれる。

1. 中心点候補を環境条件でふるい落とす
2. その中心点の周囲で、半径ごとに円周上の観測点 3 点を探す

本版の特徴は、距離計算を Web Mercator ではなくローカル UTM で処理する点にある。これは半径、静穏距離、道路距離といったメートルベース条件を扱うには妥当な改善である。

## 2. スクリプト構成

スクリプトは大きく次のブロックで構成される。

1. 入力ファイル名と制御パラメータの定義
2. 半径リストと色設定
3. ローカル UTM 投影の決定
4. シード読込と座標変換
5. 出力 CSV 初期化
6. ヘルパ関数群
7. シードごとの OSM 取得
8. 中心点探索
9. 半径ごとのアレイ探索
10. CSV / 地図保存

## 3. 入力ファイルの意味

### 3.1 `centers.csv`

列は `id, lat, lon`。ここで与えられた点は「最終中心点」ではなく、探索の起点となる seed である。

この seed の周囲 `CENTER_SEARCH_RADIUS` m の円内で、さらに中心点候補を探す。

今回ディレクトリ上の `centers.csv` には `MYZ` の 1 点が入っていた。

### 3.2 `radius.csv`

探索したい観測半径の一覧。読み込んだ値は昇順ソートされ、10 個以上だと例外になる。

今回確認した値:

- 200
- 250
- 400
- 500
- 700
- 1000
- 1200

### 3.3 `ctrl-param.csv`

探索パラメータとペナルティ重みを保持する設定ファイル。`#` 行は無視される。

主要パラメータ:

- `CENTER_SEARCH_RADIUS`: seed 周囲で中心候補を探す半径
- `CENTER_GRID_STEP`: 中心候補探索グリッドの間隔
- `ANGLE_STEP`: 正三角形方位の探索刻み
- `ANGLE_TOL`: 各目標方位に対する許容角度
- `QUIET_DISTANCE`: noisy から最低限確保したい距離
- `ROAD_TOL`: prefer からの許容逸脱距離
- `SEARCH_RADIUS`: OSM 取得範囲
- `RADIUS_TOL_RATIO`: 再探索時の半径ずらし許容率
- `RADIUS_REFINE_STEP`: 再探索時の半径刻み
- `MAX_CANDIDATES`: 上位中心候補の保存数
- `PENALTY_WATER`, `PENALTY_NOISY_SCALE`, `PENALTY_PREFER_SCALE`: 環境ペナルティ重み

## 4. ローカル CRS 化の意味

旧版 `microtremor_array_search202501-v4.py` は投影先に `EPSG:3857` を使っていた。これは広域可視化には便利だが、距離の厳密評価用途では適切でない場合がある。

本版では:

- seed 群の平均緯度経度を求める
- そこから UTM ゾーンを決める
- `PROJ_CRS` を生成する
- 以後の幾何計算をその CRS 上で実施する

これにより、以下の計算がメートルベースで自然になる。

- 中心候補の探索半径
- noisy からの距離判定
- prefer からの距離判定
- 円周生成 `center.buffer(R)`
- 観測点半径誤差

## 5. OSM データの意味づけ

スクリプトは OSM データを用途別に 4 種類へ分けている。現行版は `ctrl-param.csv` の `OSM_SOURCE` で、`local_pbf` と `overpass` を切り替える。

### 5.1 `noisy`

騒音または振動リスクが高いと見なす対象。

- `highway = motorway, trunk, primary`
- `railway = True`

これに近い中心点・観測点は不利になる。

### 5.2 `prefer`

観測点設置候補として好ましい対象。

- `leisure = park`
- `landuse = recreation_ground`
- `highway = residential, service, unclassified, track, path, footway, cycleway, tertiary, pedestrian`

観測点は基本的にこの近辺に乗る前提で設計される。

### 5.3 `fallback`

`secondary` 道路。`prefer` で交点が乏しい場合の補助候補として扱われる。

### 5.4 `water`

水域または水路。中心点は水域上だと不採用、観測点は極大ペナルティ対象。

## 6. 中心点探索の詳細

### 6.1 グリッド探索

seed の周囲 `CENTER_SEARCH_RADIUS` m の円内に対し、`CENTER_GRID_STEP` m 間隔で格子点を打つ。

ただし、格子点そのものを直接中心点にするのではない。各格子点の周囲 `CENTER_GRID_STEP / 2` m の範囲をさらにサンプリングし、その中から条件を満たす点を 1 つ拾う。

### 6.2 `sample_center_candidates()`

この関数は格子点の周囲を細かく走査して候補点を生成する。`SEARCH_R = CENTER_GRID_STEP / 2`, `SAMPLE_STEP = SEARCH_R / 2` としているため、実質的には 1 格子点のまわりに粗すぎず細かすぎない小さなローカル探索を行っている。

### 6.3 `is_center_ok()`

中心点候補が有効かは次で判定される。

- 水域上ではない
- noisy から `QUIET_DISTANCE` m 以上離れる
- prefer から `ROAD_TOL` m 以内

つまり中心点は「静かで、かつ道路/公園に近い場所」に限定される。

### 6.4 実装上の癖

1 つのグリッド点に対して、条件を満たす候補点が複数あっても最初に見つかった点だけを採用する。したがって、ここでは最適化ではなく早期打ち切りが行われている。

## 7. 観測点探索の詳細

### 7.1 円周との交点

半径 `R` の観測点を探す際、中心点 `c0` の周りに半径 `R` の円周を作り、その円周と `prefer_u`, `fallback_u` の交差を取る。

交差結果の扱い:

- 点ならそのまま採用
- 複数点なら全部採用
- 線なら 20 点に等分補間して採用

要するに「円周上で道路や公園境界と接する位置」を候補観測点としている。

### 7.2 正三角形探索

`theta = 0..359` を `ANGLE_STEP` で走査し、目標方位:

- `theta`
- `theta + 120`
- `theta + 240`

に対して、各角度帯 `ANGLE_TOL` 以内の候補点を探す。

3 方向そろえば、それを 1 つの候補アレイとして誤差評価する。

### 7.3 `triangle_error()`

評価値は次の 3 項の和である。

1. 半径誤差
2. 角度誤差
3. 観測点環境ペナルティ

半径誤差は「円周からのズレ」、角度誤差は「120 度間隔からのズレ」、環境ペナルティは「水域、騒音源、prefer 逸脱」による不利を表す。

## 8. 候補のランキング

このスクリプトにはランキングが 2 系統ある。

### 8.1 上位中心候補 `top`

`scored.append(((len(valid), max(valid)), c0, valid))` となっているため、優先順は:

1. 成立した半径の数が多い
2. 成立した最大半径が大きい

であり、誤差最小ではない。

したがって `array_primary_results.csv` は「誤差が最も小さい中心」ではなく、「多くの半径に対応できる中心」を優先して出力する。

### 8.2 半径別ベスト `best_by_radius`

一方、`best_by_radius[R]` は各半径ごとに誤差最小の中心点・観測点を保持する。

このため:

- `array_primary_results.csv`
- `array_best_by_radius.csv`

は同じ中心点群にならないことがある。

## 9. 再探索ロジック

`top[0]` の中心点で不足半径があった場合のみ、`R0 ± dR` を順に試して代替半径を探す。

探索順は:

- `R0`
- `R0 + step`
- `R0 - step`
- `R0 + 2*step`
- `R0 - 2*step`
- ...

ただし、この再探索結果は `valid` には入るが `best_by_radius` には反映されない。したがって:

- 上位候補表示には出る
- 半径別ベストには出ない

という不整合が起こり得る。

## 10. 出力ファイルの読み方

### 10.1 `array_primary_results.csv`

上位中心候補ごとに、半径別アレイを 4 行単位で近い形で保持する。

- 中心点 1 行
- 観測点 3 行

列の意味:

- `seed_id`: seed 名
- `rank`: 上位候補順位
- `radius`: 設計半径
- `center_lat`, `center_lon`: そのアレイの中心
- `point_id`: 0 が中心、1..3 が観測点
- `lat`, `lon`: その行の点座標
- `azimuth_deg`: 観測点の方位
- `triangle_error`: そのアレイの評価値

### 10.2 `array_best_by_radius.csv`

半径別最適アレイを保存する。中心点は `triangle_error = 0` で保存されるため、この値だけでは中心行と観測点行の評価を比較しない方がよい。

### 10.3 `array_penalty_breakdown.csv`

半径別ベストアレイについて、各点の環境ペナルティ内訳が入る。観測点だけでなく中心点も保存する点が重要。

## 11. 今回確認した既存出力

ディレクトリ上の既存出力を目視確認した範囲では、`MYZ` に対して半径ごとの結果が出ている。

確認できたこと:

- `array_primary_results.csv` には rank=1 の結果が並ぶ
- `array_best_by_radius.csv` には半径ごとに観測点 3 行 + 中心点 1 行がある
- `array_penalty_breakdown.csv` の先頭部は全点で 0 ペナルティ

このことから、少なくともそのケースでは道路/公園近傍かつ noisy/water 条件を満たす解が取れている。

## 12. 技術的評価

### 12.1 良い点

- ローカル UTM 化は妥当
- OSM タグの役割分担が明確
- 半径別最良解と上位中心候補を別管理している
- ペナルティ内訳 CSV があり、後工程の診断に使いやすい

### 12.2 弱い点

- 中心点選定が最適化ではなく先着
- 観測点選定も角度帯内の先着
- `top` の順位付けが誤差を見ていない
- 再探索が `best_by_radius` に反映されない
- `MAX_CANDIDATES` は実質 3 以下前提

## 13. 改修時の優先論点

優先度高:

1. 再探索結果を `best_by_radius` に反映するか仕様確定する
2. `top` のランキング指標に誤差を含めるか決める
3. `find_array()` で角度帯内候補の組合せ探索または最近角優先を導入する

優先度中:

1. 中心点候補を最小ペナルティまたは代表点として選び直す
2. `MAX_CANDIDATES` の描画安全性を上げる
3. ログや出力に「再探索で代替半径を使ったか」を明示する

優先度低:

1. 未使用 import の整理
2. 関数分割とテスト容易性の改善

## 15. GeoFabrik ローカル PBF 対応

Git 管理側の現行版では、GeoFabrik などから取得した `.osm.pbf` をローカル読込する経路を実装済みである。

実装要点:

- `OSM_SOURCE=local_pbf` を追加
- `OSM_PBF_FILE`, `LOCAL_OSM_MARGIN`, `LOCAL_CACHE_DIR`, `OGR2OGR_BIN` を追加
- `pyogrio` で `lines` / `multipolygons` を読み、タグ分類で `noisy / prefer / fallback / water` を構成
- 直接 PBF を毎回読む代わりに、seed 周辺 bbox を `ogr2ogr` で `local_osm_cache/*.gpkg` に切り出して再利用
- `OSM_SOURCE=overpass` の従来経路は後方互換として残す

この構成により、既定運用では Overpass 通信なしで再現実行できる。

## 17. 結論

このスクリプトは、OSM の道路・公園・水域・幹線交通情報を使って、実地観測に使いやすい 3 点アレイを半自動で選ぶ実務寄りの探索ツールである。

本版の本質的な改善は「距離計算をローカル UTM に移したこと」であり、これは設計意図に対して妥当である。一方で、中心点選定と観測点選定には「最初に見つかった可用点を採る」癖が残っており、厳密な最適化というよりヒューリスティック探索に近い。

今後の改修では、再探索の整合性、ランキング指標、角度帯内候補の比較方法を優先的に見直すべきである。
